# Глава 8: Ръководство за инсталация и използване

Настоящата глава описва необходимите стъпки за стартиране и използване на платформата Odisea. Целта е работата със системата да бъде възпроизводима както в среда за разработка, така и при контейнеризирано стартиране. Описани са основните зависимости, стартирането на клиентската и сървърната част, конфигурацията на базата данни и основните действия на потребителите.

## 8.1 Необходими зависимости

За стартиране на системата чрез контейнери е необходим Docker с Docker Compose. Този подход е най-лесен, защото базата данни, сървърният интерфейс и клиентската част се стартират като предварително описани услуги.

За локална разработка са необходими и следните инструменти:

| Инструмент     | Предназначение                                    |
| -------------- | ------------------------------------------------- |
| .NET SDK       | Стартиране и разработка на сървърната част        |
| Node.js        | Стартиране и изграждане на клиентските приложения |
| Git            | Управление на версиите и работа с хранилището     |
| Docker         | Стартиране на инфраструктурата в контейнери       |
| Docker Compose | Оркестрация на услугите                           |

Таблица 8.1. Основни зависимости за стартиране на проекта

При контейнеризирано стартиране PostgreSQL не е необходимо да бъде инсталиран отделно, тъй като се стартира като самостоятелна услуга чрез Docker Compose.

## 8.2 Стартиране на клиентската част

Клиентската част включва административния портал, публичния сайт и библиотеката с вградими компоненти. В режим на разработка порталът се стартира чрез Node.js и локален development server. Това позволява автоматично презареждане при промени в кода и улеснява ежедневната разработка.

Примерна последователност за стартиране на портала е:

```bash
cd frontend/portal
npm install
npm run start
```

Библиотеката с вградими компоненти се намира в отделна директория и се компилира до самостоятелен пакет. Този пакет може да бъде използван в сайтовете на агенциите чрез предоставения код за вграждане.

```bash
cd frontend/components
npm install
npm run build
```

При контейнеризирано стартиране клиентската част се изгражда предварително и се отдава от уеб сървър. Това доближава поведението до реална среда за внедряване.

## 8.3 Стартиране на сървърната част

Сървърната част се намира в директорията `backend`. В режим на разработка тя може да бъде стартирана чрез инструментите на .NET.

```bash
cd backend/src/Odisea.WebAPI
dotnet run
```

При първо стартиране се прилагат миграциите към базата данни и се зарежда начална конфигурация, ако такава е предвидена в проекта. Това осигурява необходимата структура на таблиците и позволява системата да започне работа без ръчно създаване на базата.

При стартиране чрез Docker тези действия се изпълняват в контейнера на сървърния интерфейс. По този начин локалната настройка се свежда до стартиране на контейнерите.

## 8.4 Конфигурация на базата данни

Системата използва PostgreSQL като основна база данни. При контейнеризирано стартиране базата работи като отделна услуга и използва собствен Docker том за съхранение на данните.

В конфигурацията базата е публикувана на порт `5433`, за да не влиза в конфликт с локална PostgreSQL инсталация, която често използва стандартния порт `5432`.

Примерни стойности за локална среда са:

| Параметър | Стойност                   |
| --------- | -------------------------- |
| Host      | localhost                  |
| Port      | 5433                       |
| Database  | odisea                     |
| User      | postgres                   |
| Password  | задава се в конфигурацията |

Таблица 8.2. Примерна конфигурация на базата данни

Промените по схемата се управляват чрез миграции. Това позволява структурата на базата да се развива заедно с кода и намалява риска от несъответствия между различни среди.

## 8.5 Стартиране с Docker

Най-лесният начин за стартиране на цялата система е чрез Docker Compose от основната директория на проекта.

```bash
docker compose up -d --build
```

Командата изгражда необходимите образи и стартира основните услуги: базата данни, сървърния интерфейс и клиентската част. След успешно стартиране системата е достъпна на следните адреси:

| Услуга                 | Адрес                 |
| ---------------------- | --------------------- |
| Портал и публичен сайт | http://localhost:4200 |
| Сървърен интерфейс     | http://localhost:8080 |
| PostgreSQL             | localhost:5433        |

Таблица 8.3. Локални адреси на услугите

Спирането на услугите се извършва със следната команда:

```bash
docker compose down
```

Ако е необходимо системата да бъде върната в чисто състояние, може да се премахне и томът на базата данни. Това изтрива локално съхранените данни, затова трябва да се използва внимателно.

```bash
docker compose down -v
```

## 8.6 Основни стъпки за използване

Работата с платформата започва от публичния сайт. Той представя основната идея на Odisea и насочва потребителя към регистрация или вход в системата.

![Фигура 8.1. Начална страница на платформата](screenshots/landing.png)

Нов потребител създава акаунт през регистрационната форма. При регистрация се посочват основни данни за потребителя и организацията, например име, електронна поща, име на агенция или туроператор и избран тип профил.

![Фигура 8.2. Регистрация на нов акаунт](screenshots/register.png)

След вход в системата действията зависят от ролята на потребителя. Туроператорът управлява оферти, като може да ги въвежда ръчно, групово или чрез връзка към външен източник. Агенцията работи с наличния каталог и създава колекции от оферти, които да публикува в своя сайт.

![Фигура 8.3. Управление на колекции](screenshots/collections.png)

Визуалната идентичност на публикуваните оферти се настройва чрез теми. Агенцията може да избере тема и да я адаптира спрямо своя сайт, без да променя кода на вградимите компоненти.

![Фигура 8.4. Пазар на теми](screenshots/marketplace.png)

След като колекцията, изгледът и темата са избрани, агенцията създава публикация. Системата генерира код за вграждане, който се поставя в сайта на агенцията. Когато посетител отвори страницата, офертите се зареждат чрез вградимите компоненти.

При интерес от страна на туриста се изпраща запитване. То постъпва в обща кутия в административния портал, където агенцията може да го прегледа и обработи.

![Фигура 8.5. Кутия със запитвания](screenshots/leads.png)

За връзка с външни системи агенцията може да регистрира изходящи известия. Те се използват при събития като ново запитване и позволяват интеграция с други инструменти.

![Фигура 8.6. Управление на изходящи известия](screenshots/webhooks.png)

## 8.7 Административна част

Администраторът на платформата има достъп до функции извън рамките на отделна организация. Той може да наблюдава регистрираните организации, да управлява общи настройки, да преглежда състоянието на системата и да подпомага поддръжката при възникване на проблеми.

Тази роля се използва внимателно, тъй като има по-широк достъп от ролите на агенциите и туроператорите. Основната ѝ цел е техническо и организационно управление на платформата, а не ежедневна работа с оферти и публикации.

Описаните стъпки показват как системата може да бъде стартирана, конфигурирана и използвана в основните си сценарии. Следващата глава прави оценка на постигнатото и очертава възможностите за бъдещо развитие.
